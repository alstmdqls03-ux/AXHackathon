# solution.md — 솔루션 후보 + 확정 피치

> - 대상 문제: **후보 C — 매매 의사결정 설득 에이전트 ("지금 어떻게 해?")** — 사용자 승인(`DECISION.md`, 2026-07-09). 승인 조건 = 설계 제약 4건의 **코드 구조 강제** → §5에 명세.
> - 근거 인용은 대장 번호(`sources.md` Sx-n/I-n) + 힌트 타임스탬프. 가중치: S1(출제자 힌트) 최상위.
> - 공통 산출물 형태: [① 진단 → ② 선택지 병렬 비교 → ③ 실행 체크리스트] 3단 고정 리포트 (problem.md §3·§5).

## 공통 데모 샘플 (세 후보 동일 입력 — 숫자는 합성 샘플 데이터이며 실제 시세 주장이 아님)

**입력 1** — `trades.csv` (사용자 거래 내역):
```csv
date,ticker,side,qty,price_usd,fee_usd,fx_krw_per_usd
2026-05-04,TSLA,BUY,2,180.00,0.25,1350
2026-06-10,TSLA,BUY,1,150.00,0.12,1400
```
**입력 2** — 현재 시점 파라미터(사용자/데모 제공): 현재가 $160.00, 현재 환율 1,380 KRW/USD, 매도 수수료율 0.07%.
**입력 3** — 사용자 의도(자연어): *"테슬라 파란불인데… 나 손해 본 거야? 지금이라도 팔아야 해?"* (S28-1의 실제 유저 언어 패턴)

**기대 출력** — 3단 고정 리포트 (발췌):

```markdown
# 매매 의사결정 리포트 — TSLA 3주

## ① 진단 — "손해 봤나?"의 실체
| 지표 | 값 | (metric id) |
|---|---|---|
| 명목 수익률(가격만, USD) | -5.88% | metrics.nominal_return_usd_pct |
| 실질 수익률(수수료 포함, USD) | -5.95% | metrics.true_return_usd_pct |
| **실질 수익률(원화 환산)** | **-4.90%** | metrics.true_return_krw_pct |
| 환율 효과 | +1.05%p (달러 약세가 손실을 줄여줌) | metrics.fx_effect_pp |
| 평가손익(원화) | -34,106원 | metrics.unrealized_pnl_krw |
| 지금 전량 매도 시 확정 손실(제비용 반영) | -34,569원 | metrics.realized_loss_if_sell_krw |
| 매도 시 양도소득세 | 0원 (손실 실현 — 과세 없음. 올해 다른 해외주식 이익과 손익통산 가능) | metrics.tax_if_sell_krw |
→ 요약: 앱의 파란불(-5.88%)보다 실제 원화 손실은 작다(-4.90%). 차이의 이유: 환율 +1.05%p, 수수료 -0.07%p.

## ② 선택지 비교 (가나다순 — 이 리포트는 순위·추천을 제시하지 않으며, 결정은 사용자의 몫)
### 보유 지속 | 부분 매도 | 분할 추가 매수 | 전량 매도
(각 선택지 동일 구조: [지금 일어나는 일 / 근거 수치 / 확정되는 것 / 이 리포트가 알 수 없는 것])
예: **전량 매도** — 일어나는 일: 손실 -34,569원 확정(metrics.realized_loss_if_sell_krw), 세금 0원(metrics.tax_if_sell_krw).
확정되는 것: 추가 하락 위험 제거. **알 수 없는 것: 내일의 주가 — 이 리포트는 미래 수익률을 예측하지 않는다.**

## ③ 실행 체크리스트 (사용자가 선택한 뒤 출력되는 고정 템플릿)
예: [전량 매도 선택 시] □ 주문 전 최종 확인: 수량 3주 □ 시장가/지정가의 의미 □ 체결 후 확인할 것(체결가·수수료) □ 손익통산 메모
```

## 후보 1 — SKILL.md + 결정론 계산 스크립트 + 스키마 검증기 (하이브리드) ★ 추천

- **동작 절차**: SKILL.md가 흐름을 지휘 — (1) `scripts/diagnose.py trades.csv --price 160 --fx 1380` 실행 → 진단 지표 JSON(metric id 부여) (2) LLM은 선택지별 서술을 **JSON으로만** 작성(근거 필드는 metric id 참조만 허용) (3) `scripts/render.py diagnosis.json options.json` 이 스키마 검증 후 3단 마크다운 리포트를 조립. 검증 실패 시 리포트 미생성 + 에러.
- **구현 형태**: Codex 플러그인 `src/` = `.codex-plugin/plugin.json` + `SKILL.md` + `scripts/diagnose.py`·`render.py`(표준 라이브러리만) + 샘플 데이터. MCP 서버 불필요.
- **비용·리스크**: 스크립트 2개(각 ~150줄 내외) + 스키마 + SKILL.md — 마감(내일 23:59) 내 구현·검증 여유. 리스크: 세금·수수료 규칙 단순화 필요 → 데모 범위(미국 주식 1종목, 양도세 기본 공제 룰)로 한정하고 한계를 리포트에 명시.
- **실패·정보 부족 동작** (문항 3 소재):
  1. 현재가/환율 미제공 → `REQUIRED_INPUT_MISSING: price` 에러와 함께 어떤 값이 왜 필요한지 출력 → 스킬은 사용자에게 되묻는다. **임의 추정값 삽입 금지** — 검증기가 metrics에 null이 있으면 렌더 거부.
  2. CSV 오류(음수 수량, 날짜 역순, 알 수 없는 side) → 행 번호와 함께 검증 에러 목록.
  3. 지원 외 자산·복수 종목 → `UNSUPPORTED_SCOPE` + 데모 범위 안내.
  4. 의도 모호("주식 어떡해?") → 종목·보유 여부를 묻는 질문 1개 후 진행.
  5. LLM이 근거 필드에 metric id 외 텍스트를 넣으면 → 렌더러가 `EVIDENCE_NOT_GROUNDED` 로 거부(예외 로그 소재).

## 후보 2 — 순수 SKILL.md (프롬프트 온리)

- **동작 절차**: 동일 샘플 입력을 SKILL.md 지시문만으로 처리 — LLM이 직접 손익을 암산하고 리포트 마크다운을 직접 작성. 출력 형식은 같은 3단 템플릿을 "지시"로 요구.
- **구현 형태**: `src/` = plugin.json + SKILL.md. 코드 0줄.
- **비용·리스크**: 구현 비용 최소(수 시간). 그러나 (a) 산술 오류 위험 — 환율·수수료 복합 계산을 LLM 암산에 맡기면 숫자 하나가 틀려도 '설득 리포트' 전체 신뢰 붕괴(렌즈 2 판정단 지적과 동일), (b) **DECISION.md 승인 조건 미충족** — 서열화 금지·근거 접지·고정 템플릿이 전부 프롬프트 권고 수준에 머묾, (c) 정상/예외 로그로 "스스로 증명"(S1-10)할 결정론 표면이 없음.
- **실패·정보 부족 동작**: 프롬프트로 "정보 없으면 물어라"를 지시할 수 있으나 강제 불가 — LLM이 임의 가정으로 계산할 위험을 구조적으로 막지 못함.

## 후보 3 — 로컬 MCP 서버

- **동작 절차**: 후보 1과 동일한 로직을 MCP 도구(`diagnose`, `compose_options`, `render_report`)로 노출 — Codex가 도구 호출로만 리포트를 만들 수 있어 강제력은 후보 1과 동등 이상.
- **구현 형태**: `src/` = plugin.json + SKILL.md + `.mcp.json` + MCP 서버 코드(Python) — 서버 기동·등록·수명 관리 필요.
- **비용·리스크**: 강제력은 좋으나 이동 부품 최다 — 서버 기동 실패·등록 문제가 "평가 시 직접 구동" 요건의 리스크. 같은 강제력을 후보 1이 스크립트 2개로 얻는다(과잉 설계, ~2주 스코프 원칙 위반 방향).
- **실패·정보 부족 동작**: 후보 1과 동일 로직 + 서버 다운이라는 실패 모드가 추가됨.

## 4. 확정 피치 — 후보 1

- **클러치 피처 한 문장**: *초보 투자자의 "지금 어떡해?"에 대해, 결정론 계산으로 손익의 실체를 진단하고 — 순위 없는 선택지 비교와 실행 체크리스트로 — 사용자가 납득하고 스스로 결정하게 만드는 매매 의사결정 리포트 생성 플러그인.*
- **왜 이 방식이 가장 잘 푸는가**: 출제자의 평가 기준은 "적절한 해답 + 사용자가 충분히 납득해서 안심하고 그다음을 실행"(S1-8 [01:41])과 "해답을 만들어 나가는 과정의 설계"(S1-9 [01:51])다. 후보 1은 이 사슬을 코드 구조로 강제한다 — 진단(납득의 재료)은 결정론 계산으로 틀릴 수 없게, 선택지(안심의 구조)는 서열 없는 병렬 비교로, 실행(그다음)은 체크리스트로. "수학적 최적점이 아니다"(S1-7 [01:32])라는 부정과도 정합: 계산은 설득의 재료일 뿐, 리포트는 미래를 예측하지 않으며 결정을 대신하지 않는다(금융위 보조수단성 S10-1). 형상 없는 에이전트 비전(S1-15 [03:29]~[04:02])과 출제자 본인의 Codex 사용(S1-14 [03:11])에 형태가 일치하고, "AI로 해결 방법을 찾고 그게 맞다는 걸 스스로 증명"(S1-10 [02:00])은 결정론 스크립트의 정상/예외 실행 로그로 입증한다.
- **데모 계획**: ① 정상 실행 — 위 샘플 입력 → 3단 리포트 생성 (비주얼: 고정 템플릿 마크다운 리포트 — 진단 표 + 선택지 4개 병렬 카드 + 체크리스트). ② 예외 1 — 현재가 누락 → `REQUIRED_INPUT_MISSING` + 되묻기. ③ 예외 2 — 근거 접지 위반 주입 → `EVIDENCE_NOT_GROUNDED` 거부. 세 실행 모두 이 폴더 세션에서 수행해 로그에 남김(문항 5 정합).

## 5. 승인 조건 이행 — 설계 제약 4건의 코드 구조 강제 (DECISION.md)

| # | 제약 | 강제하는 코드/스키마 구조 |
|---|---|---|
| 1 | 선택지 서열화 금지 | 리포트 JSON 스키마의 `options[]`에 **순위·점수·추천 필드 자체가 존재하지 않음**(additionalProperties: false). 렌더러가 선택지를 **가나다순으로 강제 정렬**해 LLM이 배치 순서로도 서열을 표현할 수 없음. 모든 선택지는 동일 4필드 구조([일어나는 일/근거/확정되는 것/알 수 없는 것])로 렌더링. |
| 2 | 근거는 진단 계산 산출물에만 접지 | `options[].evidence[]`는 `^metrics\.[a-z_]+$` 패턴 + **diagnose.py 출력에 실존하는 metric id**만 통과. 자유 텍스트·외부 데이터·시세 예측이 근거 필드에 들어오면 렌더러가 `EVIDENCE_NOT_GROUNDED`로 리포트 생성 거부. |
| 3 | 3단 고정 템플릿 | 최종 마크다운은 LLM이 아니라 **render.py가 조립** — 섹션 구성([진단→선택지→체크리스트])·표 형식·경고 문구("순위·추천을 제시하지 않는다", "미래 주가를 알 수 없다")가 코드에 하드코딩. LLM 산출물은 스키마 검증을 통과한 JSON 필드 값으로만 유입. |
| 4 | 공백 주장 (추정) 유지 | 코드가 아닌 문서 규율: README·question.md에서 I-1(로드맵 공백)·I-2(업계 공백)는 (추정) 표기로만 인용하고 차별화 본체는 [01:41] 사슬의 구조적 구현에 둠 — step-05+ 문서 게이트에서 체크. |

- 한계의 정직한 명시: Codex(LLM)가 스크립트를 우회해 직접 텍스트를 낼 가능성은 프롬프트만으로 0이 되지 않는다. 그러나 **제출물의 산출 경로가 "스크립트 실행 → 검증 → 렌더"로 고정**되어 있고, 우회 시 리포트 형식 자체가 만들어지지 않으므로(렌더러 출력만이 유효 산출물) 위반은 로그에서 즉시 드러난다 — "스스로 증명"(S1-10) 프레임과 일치.
