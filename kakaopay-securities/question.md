## 문항 1 — 무엇을, 누가, 어떤 상황에서 쓰나요? (≤800자)

trade-decision-report는 초보 투자자가 보유 종목 앞에서 "나 손해 본 거야? 지금 어떡해?"라고 물을 때, 결정론 계산으로 실질 손익을 진단하고 순위 없는 선택지 비교와 실행 체크리스트를 담은 매매 의사결정 리포트를 만들어 주는 Codex 플러그인입니다.

쓰는 사람은 카카오페이증권의 핵심 고객층인 초보 투자자입니다(종합계좌 700만 개, 20~40대 비중 70%대 — 회사 공식 인터뷰). 상황은 이렇습니다. 해외주식이 파란불인데 앱의 수익률 숫자는 가격 기준 명목치일 뿐, 수수료·환율·세금까지 반영하면 내 실제 손익이 얼마인지, 왜 그런지는 아무도 설명해 주지 않습니다. 커뮤니티에 물으면 존버·물타기·손절로 답이 엇갈려, 납득할 근거 없이 앱만 하루에도 수십 번 열어보게 됩니다.

플러그인은 거래 내역 CSV와 현재가·환율을 받아 ①진단(명목 vs 실질 수익률, 환율 효과 분해, 지금 팔면 확정되는 손익·세금) ②선택지 4종(보유·부분 매도·분할 추가 매수·전량 매도)의 병렬 비교 — 각각 "일어나는 일/근거 수치/확정되는 것/알 수 없는 것" ③선택한 행동의 실행 체크리스트, 이 3단 고정 리포트를 생성합니다. 순위와 추천은 구조적으로 불가능하게 만들었고 결정은 항상 사용자의 몫입니다. 이 플러그인이 주는 것은 정답이 아니라 납득 과정이며, 사용자가 안심하고 다음 행동을 실행하게 하는 것이 목적입니다.

## 문항 2 — 왜 이 문제를 선택했나요? (≤800자)

출제 기업이 직접 지목한 문제이기 때문입니다. 카카오페이증권 AI서비스센터장은 힌트 영상에서 "초보 투자자들이 매수 매도를 할 때 내가 손해 보지 않았다라든지, 어떻게 해야 될지 잘 모르겠는데 어떻게 도움을 받을 수 있을까에 대한 니즈가 있다… 많은 분들이 괴로워한다는 생각이 들어서 이 문제를 냈습니다"라고 말했습니다(영상 01:03~01:19).

이 고통은 공개 데이터로 실증됩니다. 자본시장연구원 분석(개인투자자 약 20.4만 명)에서 2020년 신규 투자자의 60%가 손실을 봤고(거래비용 차감 후 -1.2%), 연간 거래회전율은 1,600%를 넘었으며, 이익 난 주식을 손실 주식보다 2배 이상 먼저 파는 처분효과(보유 1일 기준 41% vs 22%)가 확인됩니다. 유저의 실제 언어도 같습니다 — 공개 Q&A에는 "-20%인데 물타기 알려주세요"라는 질문에 존버/분할매수/손절로 엇갈린 답만 달립니다. 회사도 이를 공식 인정합니다: "많은 사용자들이 언제 사고 팔아야 할지 판단하기 어려워 수익 실현 타이밍을 놓치는 경우가 많았다"(수익률 모으기·팔기 출시 배경).

즉 문제의 본질은 정보 부족이 아니라(출제자: "접할 수 있는 정보는 거의 다 동일") 자기 상황에 적용해 납득시켜 주는 설명의 부재입니다. 임계값 자동화까지 나온 지금, "왜"를 채워 사용자를 안심시키는 계층이 주식 모으기 이용자 160만 명으로 대표되는 초보 투자자층에 걸린 빈칸이라고 판단했습니다.

## 문항 3 — 플러그인은 어떻게 작동하나요? (≤1000자)

3단계 파이프라인이고, LLM이 글을 쓰는 구간은 하나뿐입니다.

①진단(diagnose.py): 거래 내역 CSV와 현재가·환율을 받아 실질 손익을 결정론 계산합니다 — 명목 vs 실질 수익률, 환율 효과 분해(평균 매수 환율 대비 강세/약세), 지금 팔면 확정되는 손익과 양도소득세(단순화). 시세·뉴스는 조회하지 않으며 모든 숫자는 입력에서 유도됩니다.

②선택지 작성(LLM): 진단을 보고 4개 선택지의 "일어나는 일/근거/확정되는 것/알 수 없는 것"을 JSON으로 씁니다.

③검증·조립(render.py): JSON 검증 후 고정 템플릿 마크다운을 조립해 report.md로 저장합니다(입력 해시 기반 report-id 스탬프 = 유효 산출물의 정의).

판단 기준은 프롬프트 권고가 아니라 코드로 강제됩니다. 순위·추천 불가 — 선택지는 4종 고정(제목은 렌더러가 하드코딩), 스키마에 순위 필드가 없고 가나다순 강제 정렬, 추천·비교·예측 어휘는 거부(RANKING_LANGUAGE_DETECTED). 모든 숫자는 계산 산출물 — 근거 필드는 실존 진단 지표 참조만(EVIDENCE_NOT_GROUNDED), 서술 속 숫자는 {metrics.id} 플레이스홀더만 허용, 숫자 리터럴은 거부(UNGROUNDED_NUMBER).

정보가 부족하거나 안 풀리는 상황: 현재가·환율이 없으면 어떤 값이 왜 필요한지 출력하고 되묻습니다 — 임의 추정값은 대입하지 않습니다(REQUIRED_INPUT_MISSING). CSV 오류는 행 번호별로 알려주고, SELL 내역·복수 종목 등 범위 밖은 설명하며 거부합니다(UNSUPPORTED_SCOPE). LLM 산출물이 규칙을 어기면 리포트를 만들지 않고 위반을 공개한 뒤 재작성합니다. 추천을 요구받으면 순위를 내지 않는 이유를 설명하고 판단 재료를 안내합니다. 한계 고지: 세금은 단순화 계산이며 이 플러그인은 투자 자문이 아니고 미래 주가를 알 수 없습니다. 세부는 zip의 README.md와 동일합니다.

## 문항 4 — AI를 어떻게 썼나요? (≤800자)

AI에게 맡긴 것: 힌트 영상 전사·구조화, 공개 자료 리서치(모든 출처를 검증 에이전트가 직접 열어 로그인 불요·수치 일치 확인), 코드·문서 초안 작성. 문제 선정과 스펙은 서로 다른 관점의 AI 판정단이 독립 채점하고 서로 반박하게 했습니다.

직접 판단한 것: 문제 최종 선정(후보 3개 중 "매매 의사결정 설득 에이전트"), 솔루션 형태 확정, 그리고 승인 조건 — "서열화 금지·근거 접지·고정 템플릿을 프롬프트 권고가 아니라 코드 구조로 강제하라"는 요구는 사람의 결정이었습니다.

막혔던 지점과 해결: ①스펙의 데모 수치에서 환율 효과 방향이 반대로 서술돼 있었습니다("달러 약세"라고 썼지만 평균 매수 환율 1,364.7원 대비 현재 1,380원 = 강세). 산술 전용 검증 AI가 손계산 대조로 잡아냈고, 코드에 방향 분기 로직으로 반영했습니다. ②검증 단계에서 Codex 계정 사용량 한도로 모델 턴 실행이 막혀, 설치 캐시본에서 스크립트 파이프라인을 완주하는 방식으로 대체했습니다.

받아들이지 않은 AI 제안: MCP 서버 아키텍처(강제력은 같은데 서버 기동·등록 등 이동 부품만 늘어 "평가 시 직접 구동" 리스크 증가)와 순수 SKILL.md 프롬프트 온리(구현은 가장 쉽지만 LLM 암산 오류와 추천 표현을 구조적으로 막지 못함)를 기각하고, 스크립트+스키마 검증 하이브리드를 채택했습니다.

## 문항 5 — 어떻게 검증했나요? (≤800자)

정상 경로: 샘플 거래 내역(테슬라 2회 분할 매수, 수수료·체결 환율 포함)에 현재가 160달러·환율 1,380원을 넣으면 진단 7개 지표가 스펙 기대값과 전항 일치했습니다 — 명목 -5.88% vs 원화 실질 -4.90%(환율 효과 +1.05%p), 지금 팔면 -34,569원 확정·세금 0원. 소스 트리와 설치 캐시본(심사자가 받는 경로) 양쪽에서 실행해 report-id가 동일함(8d1a510771e6)을 확인했습니다(결정론 재현).

예외 경로 3종: ①현재가 누락 → REQUIRED_INPUT_MISSING과 "왜 필요한지"를 출력하고 되묻기(임의 값 대입 없음) ②추천 어휘("가장")와 근거 없는 숫자(-40,000원)를 주입한 선택지 JSON → RANKING_LANGUAGE_DETECTED와 UNGROUNDED_NUMBER로 거부되고 리포트 파일이 생성되지 않음 ③SELL 행 포함 CSV → UNSUPPORTED_SCOPE로 범위를 안내하며 거부.

의심하고 고친 것: 스펙 데모의 환율 효과 방향 오류(약세→강세)를 산술 재검증으로 잡아 고쳤고, 리포트 조사 오류도 잡아 재렌더로 확인했습니다.

아직 부족한 것: 세금이 단순화 계산이고(손익통산·환차익 미반영), 추천 어휘 denylist는 완전하지 않으며(거부→재작성 루프로 보완), Codex 모델 턴만 계정 한도로 못 돌렸습니다. 다만 설치·활성화와 스크립트 파이프라인 완주는 검증했고, LLM 산출물은 무엇이 오든 검증이 걸러내므로 신뢰는 모델 실행에 의존하지 않습니다. 실행 기록은 전부 logs/에 있습니다.
