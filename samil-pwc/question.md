## 문항 1 — 무엇을, 누가, 어떤 상황에서 쓰나요? (≤800자) — 544자

흩어진 결산 데이터(시산표·전표·월별 손익과 현금흐름·채권연령·원가배부표)를 받아, 회계 전문가의 가설 라이브러리를 따라 교차 대조 추론을 수행하고 발견마다 원본 근거(전표ID·계정·월·제품)를 인용한 경영진 보고서를 만들어 주는 Codex 플러그인 'haenggan-audit(행간 감사)'입니다.
쓰는 사람: 삼일PwC의 회계사·컨설턴트(감사·자문 현장에서 고객 결산 패키지를 받았을 때)와 그 고객사의 CFO·재무팀·경영진.
상황: 결산 마감, 이사회 보고, 감사 대응 전에 "숨은 문제가 없는가"를 확인해야 하는데, ERP·회계·자금·원가 등 시스템별 보고서는 각각 정상으로 보입니다. 문제는 시트·계정·기간을 가로질러 대조해야만 드러납니다 — 틀린 회계처리(비용의 자산화 등), 흑자 속 현금흐름 악화, 원가배분 왜곡으로 겉으론 흑자인 손실 제품. 사람이 하려면 회계사급 전문가가 결산 패키지 전체를 수 주간 전수 교차 검토해야 해서, 실무에선 표본 검토로 넘어가고 문제는 정정 공시·감리 지적 단계에서야 발견됩니다. 이 플러그인은 그 '행간 읽기'를 대체합니다.

## 문항 2 — 왜 이 문제를 선택했나요? (≤800자) — 619자

출제 기업이 직접 준 힌트를 정조준했습니다. 삼일PwC AX Node 조홍래 파트너가 인터뷰 영상에서 과제를 정의했습니다 — "기업에 산재된 데이터를 총망라해 기업의 고민을 뽑아내는 에이전트"(Trusted CEO Agent), 예시는 틀린 회계처리·현금흐름 문제·원가배분 왜곡, 본질은 "데이터의 행간 속에 묻힌 요소 찾기". 세 예시는 별개 과제가 아니라 한 문제의 세 증상이므로 가설군 3종으로 통합했습니다.
실무자·고객이 막히는 지점은 공개 통계로 입증됩니다. 개별 숫자가 정상처럼 보여 교차 검토 없이는 못 잡고, 그래서 사후에야 드러납니다 — 상장사 131사(5.1%)가 한 해 동안 감사보고서를 정정했고(총 379회, 재무제표 본문 정정 72%), 2025 회계연도 계속기업 불확실성 '주의'가 66곳, 금감원은 분식위험지표가 높은 회사를 감리 표본으로 우선 선정합니다. 삼일PwC 스스로도 같은 카테고리에 투자 중이며(빅스데이터와 'AI 기반 경영 인텔리전스 에이전트' 공동개발 협약), PwC 글로벌 CEO 설문에서는 AI 성과를 매출·비용 양면에서 체감한 CEO가 12%(한국 14%)에 그칩니다 — '근거를 인용해 신뢰를 만드는(Trusted)' 에이전트가 정확히 그 공백을 겨냥합니다.

## 문항 3 — 플러그인은 어떻게 작동하나요? (≤1000자) — 940자

절차: ①데이터 인벤토리 — 폴더의 CSV를 파일 계약(시산표·전표·월별 시계열·채권연령·원가배부표)과 대조 ②결정론 체커(checks.py, 파이썬 표준 라이브러리만) 실행 — 시산표 산술·차대균형 검산, 기말 대형 전표·작성자=승인자·수선성 지출의 자산화 패턴 스캔, 이익-현금흐름 괴리·회수기간(DSO) 추세·분기말 스파이크·거래처 편중 계산, 대안 드라이버 원가 재배부 → 이상 후보 JSON 산출 ③에이전트가 가설 모듈 3종(H1 회계처리 오류·H2 현금흐름 이상·H3 원가배분 왜곡)을 로드해 후보를 원본 데이터와 대조 ④오탐 기각(기각도 기록) 및 신호 연결 — 예: 수선비 전기比 -78% × 유형자산 기말 대형 전표 = 자본화 의심. 이 교차가 '행간'입니다 ⑤심각도·확신도 판정 ⑥경영진 SOP 보고서 생성(행간→근거→판단 기준→재무 영향→권고→확신도 구조).
지식: 가설 모듈에 내장 — K-IFRS 1016 후속원가(수선비 자본화 요건), 기간 귀속, 승인 분리 통제, 이익의 질(발생액-현금 괴리), DSO·밀어내기 패턴, 원가 드라이버 인과성.
판단 기준: 심각도(高=손익·의사결정 반전, 기준 위반 소지), 확신도(高=데이터만으로 입증, 中 이하는 '다음 확인 절차' 필수 첨부), 인용 규칙(모든 발견은 전표ID·계정·월·제품 좌표를 인용, 인용 불가한 발견은 출력 금지 — 환각 차단).
정보 부족·실패 시: 파일이 없으면 해당 가설군만 '검토 불가(사유)'로 명시하고 나머지는 정상 수행(모듈 독립), 스키마가 다르면 동의어 매핑 후 1회만 질문, 발견 0건이면 억지 발견 없이 '검토한 가설×통과 근거'를 보고, 체커 실행 실패 시 가설 모듈의 체크 정의를 따라 에이전트가 직접 계산하는 수동 모드로 폴백(보고서에 명시), 전표가 발췌본이면 결론을 '제공 범위 내'로 한정합니다. 세부 내용은 zip의 README.md에 동일하게 적혀 있습니다.

## 문항 4 — AI를 어떻게 썼나요? (≤800자) — 720자

AI(Claude Code 오케스트레이션)에 맡긴 작업: 힌트 영상 전사·타임스탬프 정리, 공개 근거 수집과 URL 접근성 일괄 검증, 합성 데모 데이터 생성, checks.py 작성, SKILL.md·가설 모듈 문서화, 심사자 절차 재현 테스트. 제가 직접 판단한 것: ①문제 선정 — 영상의 예시 3종을 각각 풀지 통합할지 갈림길에서 '한 문제의 세 증상'으로 보고 가설군 통합안을 승인 ②아키텍처 — 순수 스킬/MCP 서버/하이브리드 중 '계산은 결정론 스크립트, 판단은 AI' 하이브리드 선택 ③스코프 가드 — 가설군당 심은 문제 1개 제한, 일정 리스크 대비 축소 경로(H2·H3 파일 삭제→단일 가설군) 사전 설계. 막혔던 지점과 해결: (a)감사보고서 정정 통계의 출처가 봇 차단(403)이라 AI 심사자가 접근 불가 → 금감원 보도자료의 KDI 공공 미러로 교체 (b)README의 플러그인 설치 명령이 실제 CLI 규격과 달랐음 → 직접 구동 테스트에서 발견해 marketplace 한정 형식으로 교정 (c)Codex 계정 사용 한도로 E2E 실행이 중단됨 → 체커 계층 검증과 스킬 트리거 확인으로 분리해 검증하는 방식으로 대응. 받아들이지 않은 AI 제안: MCP 서버 구성(심사 환경에서 서버 기동 실패 시 0점 리스크, 과잉 설계)과 대시보드형 요약 출력(출제자가 '자료 정리는 AI가 아니어도 가능'이라 감점을 명시) — 근거 인용형 보고서 구조로 대체했습니다.
