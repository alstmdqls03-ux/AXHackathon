# H1 — 회계처리 오류 (틀린 회계처리가 데이터 속에 숨어 있다)

## 가설

개별 전표·계정은 정상처럼 보이지만, 시산표↔전표↔전기 비교를 교차하면 잘못된 회계처리(비용의 자산화, 기간 귀속 오류, 통제 우회)가 드러난다.

## 필요 데이터

- 필수: `trial_balance.csv`, `gl_entries.csv` (전표는 발췌 샘플일 수 있음 — 전수 가정 금지)

## 체커 신호 해석 (checks.py → 후보)

| check | 신호의 의미 | 단독으로 결론? |
|---|---|---|
| `tb_arithmetic` / `tb_balance` | 장부 자체 산술 오류 — 데이터 신뢰성 문제 | 예 (발견 확정 가능) |
| `yoy_swing` | 계정 급변 — 원인 미상. **다른 신호와 결합해야 의미** | 아니오 |
| `period_end_large` | 기말 몰아치기 후보 — 정상 결산 분개(감가상각 등)도 걸림 | 아니오 |
| `self_approved` | 승인 통제 우회 — 금액·계정과 결합 시 위험 | 아니오 |
| `capitalization_keyword` | 수선성 지출의 자산 계상 — 자본화 요건 검토 필요 | 아니오 (증빙 없이는 '의심') |

## 추론 절차 (에이전트가 수행)

1. 같은 전표ID·같은 계정을 가리키는 신호들을 **하나의 사건으로 묶는다** (예: #4712가 capitalization+period_end+self_approved 3중이면 단일 발견으로 통합).
2. 급변 계정(`yoy_swing`)과 전표 신호의 **인과 연결을 시도**한다: "수선비 급감 + 유형자산에 수선성 대형 전표" → 비용→자산 대체 가설.
3. 반대 증거를 찾는다: 정상 CAPEX 전표(적요·승인 정상)와 비교, 감가상각 스케줄과의 정합.
4. 재무 영향을 계산한다: 수정 시 손익·자산 변화 (수정분개 형태로 제시).

## 판단 기준 (지식)

- **K-IFRS 1016호 유형자산 후속원가**: 일상적 수선·유지 지출은 발생 시 **비용**. 자본화는 미래 경제적 효익 증가(내용연수 연장·성능 향상)가 입증될 때만.
- 기간 귀속: 수익·비용은 발생 기간에 귀속 (기말 대형 전표는 귀속 시점 검토).
- 통제: 작성자=승인자는 그 자체로 내부통제(승인 분리) 미비 신호 — K-SOX 관점.

## 기각(오탐) 조건

- `period_end_large`가 **정기 결산 분개**(감가상각 계상, 월말 매출 인식 등 적요·패턴이 정상)면 기각한다. 단, 매출 인식 스파이크는 H2의 분기말 스파이크 신호와 겹치면 **H2로 라우팅**해 함께 서술.
- `yoy_swing`이 사업 성장·거래 증가로 설명되면(매출 증가율과 동행) 기각. 단 매출채권 급증은 H2 관점 재검토.
- 자본화 전표에 자본화 요건 충족 증빙(공사 내역: 내용연수 연장 등)이 데이터에 있으면 '발견'이 아니라 '확인 완료'로 부록에 기재.

## 리포트 필수 요소

전표ID·일자·금액·적요 원문 인용 / 결합한 신호 목록 / 적용 기준(K-IFRS 등) / 수정분개 예시와 손익 영향 / 확신도(증빙 미확인 시 '중' 이하)와 다음 확인 절차(예: 공사 내역서 징구).
