# 행간 감사 (Haenggan Audit) — 결산 데이터 숨은 문제 탐지

개별 숫자는 전부 정상처럼 보인다. 문제는 시트·계정·기간을 **가로질러 대조해야만** 보인다.
이 스킬은 ① 결정론 체커(`scripts/checks.py`)로 이상 후보를 재현 가능하게 계산하고, ② 에이전트(너)가 가설 모듈(`hypotheses/`)을 따라 후보를 검증·기각·연결해 ③ 근거를 인용한 경영진 보고서를 만든다.

**분업 원칙 — 반드시 지켜라**: 집계·검산 등 산술은 체커(또는 네가 작성·실행하는 코드)가 한다. 너의 몫은 산술이 아니라 **판단**이다: 오탐 기각, 신호 연결(행간 읽기), 회계 기준 적용, 영향 계산 검증, 권고. 암산으로 수치를 만들지 마라.

## 입력 계약

사용자가 준 데이터 폴더(이하 `<DATA>`) 안의 CSV (UTF-8, 첫 행 헤더):

| 파일 | 내용 | 쓰는 가설군 |
|---|---|---|
| `trial_balance.csv` | 계정코드·계정명·구분·정상잔액·기초잔액·차변합계·대변합계·기말잔액·전기잔액 | H1 |
| `gl_entries.csv` | 전표ID·일자·계정코드·계정명·차대·금액·적요·작성자·승인자 (발췌 샘플 가능) | H1 |
| `monthly_series.csv` | 월·매출·영업이익·영업활동현금흐름·매출채권잔액·재고자산잔액 | H2 |
| `ar_aging.csv` | 거래처·미회수총액·만기미도래·연체버킷들 (H2 보조 — 선택) | H2 |
| `cost_allocation.csv` | 제품·매출액·직접원가·드라이버량(노동/기계시간)·배부기준·배부액·이익 | H3 |

동봉 데모: `data/demo/` — 가상 중견 제조사 "(주)한빛산업" FY2025 (합성 데이터, 시트 간 수치 정합).

## 실행 절차

1. **인벤토리**: `<DATA>`의 파일·헤더를 확인하고 위 계약과 대조한다. 헤더가 다르면 의미가 같은 컬럼을 매핑해 보고, 확신이 없으면 사용자에게 매핑을 **한 번만** 질문한다.
2. **체커 실행**: `python3 <이 스킬 폴더>/scripts/checks.py <DATA> --out findings_candidates.json`
   - 체커는 후보(recall)를 위한 것이다. 후보 ≠ 발견. 후보 0건이어도 3~5단계는 수행한다.
   - **체커 실행 실패 시(수동 모드)**: 각 `hypotheses/H*.md`의 체크 정의와 `checks.py` 상단 임계값을 따라 동일 계산을 네가 직접 코드로 수행하고, 보고서에 "수동 계산 모드"를 명시한다.
3. **가설 모듈 로드**: `hypotheses/H1-accounting.md` `H2-cashflow.md` `H3-costing.md`를 읽는다. 각 모듈은 신호 해석 → 추론 절차 → 판단 기준 → 기각 조건 → 리포트 필수 요소를 정의한다.
4. **검증·기각·연결**: 후보마다 원본 CSV에서 근거를 재확인하고, 모듈의 기각 조건에 걸리면 기각한다(기각도 기록). 같은 사건을 가리키는 신호들은 하나의 발견으로 묶고, 가설군을 넘나드는 연결(예: H1의 채권 급증 신호 → H2 원인 분석)을 명시적으로 수행한다 — **이 연결이 '행간 읽기'다.**
5. **발견 확정**: 아래 판단 기준으로 심각도·확신도를 매긴다.
6. **보고서 작성**: `templates/report-template.md` 구조로 `report.md`를 쓴다 (한국어).
7. **자기 검산**: 보고서의 모든 수치가 (a) 체커 JSON 또는 (b) 원본 CSV 좌표 또는 (c) 명시된 계산식에서 나왔는지 확인한다. 셋 다 아닌 수치는 삭제한다.

## 판단 기준

- **심각도**: 高 = 손익·의사결정이 뒤집히거나 법규·기준 위반 소지 / 中 = 재무 영향은 제한적이나 통제·품질 문제 / 低 = 참고 사항.
- **확신도**: 高 = 결정론 계산 + 원본 재확인으로 데이터만으로 입증됨 / 中 = 정황 신호 결합, 외부 증빙 확인 필요 / 低 = 단일 신호. **확신도 中 이하는 반드시 '다음 확인 절차'를 함께 제시**한다.
- **인용 규칙(환각 금지)**: 모든 발견은 원본 좌표(전표ID·계정명·월·거래처·제품)와 수치를 인용한다. **원본에서 인용할 수 없는 발견은 쓰지 않는다.** 발견을 지어내지 않는다 — 0건이면 0건이라 보고하고, 검토한 가설과 통과 근거를 부록에 남긴다.

## 실패·정보부족 시 동작

1. **파일 누락**: 해당 가설군만 "검토 불가(사유)"로 보고서에 명시하고 나머지는 정상 수행한다 (체커도 동일하게 skip을 보고한다). 전 파일 부재면 필요한 파일 목록을 안내하고 중단한다.
2. **스키마 불일치**: 동의어 매핑 시도 → 실패 시 사용자에게 컬럼 매핑 질문(1회) → 그래도 불명확하면 해당 가설군 검토 불가 처리.
3. **발견 0건**: 정직하게 0건 보고 + 부록에 "검토한 가설 × 통과 근거". 억지 발견 금지.
4. **체커 오류·검산 불일치**: 수동 모드 폴백(위 2단계). 체커 결과와 네 재검산이 다르면 그 발견의 확신도를 낮추고 불일치 내역을 보고서 한계 절에 적는다.
5. **전표가 발췌 샘플**: 전수 검토처럼 서술하지 않는다 — "제공된 전표 범위 내"로 한정.

## 데모

```
python3 scripts/checks.py data/demo --out findings_candidates.json
```
그 후 위 절차 3~7단계 수행. 기대 결과: H1·H2·H3 각 1건의 확정 발견(+ 기각된 오탐 후보들)이 담긴 report.md.
