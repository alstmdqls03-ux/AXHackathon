## 솔루션 후보 S2 — 하이브리드: 결정적 스크립트 + LLM 판단 스킬 ★

파이프라인의 성격을 둘로 쪼갠다: **세는 일(집계·검증·리포트 조립)은 스크립트가, 생각하는 일(분류·갭 판단·초안 작성·채점)은 LLM이.** 힌트 영상의 분업론 그대로 — "AI가 작업은 다 해주고 사람이 하는 건 생각"[02:16]의 구조를 플러그인 내부에 재현하되, '생각' 단계의 프롬프트·판단 기준을 SKILL.md에 명시해 심사자가 볼 수 있게 한다[02:12 "프롬프팅…최대한 보려고"].

### 1. 데모 시나리오 (실제 입출력 예시)

**입력** — 동봉 합성 데이터(패션 커머스 가정, 60건):

```csv
# tickets.csv (핵심 컬럼)
ticket_id,created_at,question,alf_answer,resolved,agent_answer
T-1041,2026-06-02,"주문한 상품 언제 도착하나요?","평균 2~3일 소요…",true,
T-1042,2026-06-02,"산 원피스 세탁 어떻게 하나요?","죄송합니다. 답변을 찾지 못했…",false,"울코스 30도 단독세탁 부탁드려요…"
T-1043,2026-06-03,"해외 배송도 되나요?","죄송합니다…",false,"현재 일본만 가능, EMS 요금은…"
T-1044,2026-06-03,"첫구매 쿠폰이 적용이 안 돼요","쿠폰함에서 확인…",false,"세일 상품은 쿠폰 중복적용 제외라…"
T-1052,2026-06-05,"환불 계좌를 바꾸고 싶어요","고객센터로 문의…",false,"본인인증 후 변경 처리해드렸습니다"
```
```csv
# faq.csv (기존 지식 12건)
question,answer
"배송은 얼마나 걸리나요?","평균 2~3일…"
"반품 신청은 어떻게 하나요?","마이페이지 > …"
```

**동작** (4단계 — 각 단계의 판단 기준은 SKILL.md에 명문):

1. **적재·검증** `scripts/load.py`: 스키마 검사(필수 컬럼)·건수/미해결 집계 → `work/unresolved.json`. *판단 없음, 결정적.*
2. **분류·갭 도출** (LLM): 미해결 티켓을 주제 클러스터로 묶고 갭 유형 라벨링 — **A 지식 부재**(FAQ에 없음) / **B 지식 불명확**(있으나 답변 실패) / **C 정책 결정 필요**(사람만 정할 수 있음) / **D 태스크 필요**(지식으로 해결 불가, 실행 필요[E10]) / 각 판정에 confidence(높음/중간/낮음)와 근거 티켓 인용. 기존 faq.csv와 대조해 A/B를 가른다.
3. **초안 생성** (LLM): A/B 갭별 FAQ 초안 — 질문 변형 3개 + 답변(상담사 실답변 `agent_answer`가 있으면 1차 근거로 사용, 없으면 **[정책 확인 필요]** 마커를 달고 자동 확정 금지). D는 Tasks 후보로 별도 목록.
4. **재시뮬레이션 검증** `scripts/simulate.py` + LLM 채점: 미해결 티켓 각각을 "기존 지식만" vs "기존+초안" 두 조건으로 재판정(해결/부분/미해결 + 사유) → 집계는 스크립트가 수행.

**출력** — 3개 리포트 + 대시보드:

```markdown
# gap-report.md (발췌)
| 클러스터 | 건수 | 갭 유형 | confidence | 근거 티켓 |
|---|---|---|---|---|
| 상품 케어(세탁/보관) | 9 | A 지식 부재 | 높음 | T-1042, T-1047… |
| 쿠폰 적용 조건 | 7 | B 지식 불명확 | 높음 | T-1044… |
| 환불 계좌 변경 | 4 | D 태스크 필요 | 중간 | T-1052… |
```
```markdown
# validation-report.md (발췌)
시뮬레이션 해결 판정: 33/60 (55%) → 47/60 (78%)  ⬆ +14건
잔여 미해결 13건 중 6건 = C 정책 결정 필요 (사람 몫으로 이관)
※ 본 수치는 동봉 합성 데이터 기준 시뮬레이션 값
```
+ `report.html`: before/after 막대, 클러스터별 표 (외부 의존성 없는 단일 파일).

### 2. 구현 형태
- `src/.codex-plugin/plugin.json` + `src/skills/resolution-lift/SKILL.md`(단계 지침 + 판단 기준 + 프롬프트 템플릿) + `src/scripts/` (Python **표준 라이브러리만**: csv·json·html) + `examples/` 합성 데이터. MCP 불필요, 네트워크·API 키 불요.

### 3. 구현 비용과 리스크
- 비용: 1~1.5일 (스크립트 2개 소형 + SKILL.md + 합성 데이터 + 리포트 템플릿). **마감(2026-07-10) 내 가능.**
- 리스크: ①심사 환경에 python3 필요 — 표준 라이브러리만 사용 + README 명시로 완화 ②LLM 분류 변동성 — 판단 기준·라벨 체계를 SKILL.md에 고정하고 confidence 표기로 완화 ③대량 로그 토큰 초과 — 배치 분할 규칙을 SKILL.md에 내장.

### 4. 정보 부족·실패 상황 동작 (문항 3 초안)
| 상황 | 동작 |
|---|---|
| 로그 파일 없음/스키마 불일치 | `load.py`가 필수 컬럼 목록과 CSV 템플릿을 출력하고 **중단** (짐작으로 진행 금지) |
| `resolved` 컬럼 부재 | 대체 신호(`agent_answer` 존재 여부)로 미해결을 추정하되 리포트에 **(추정) 플래그** 명시 |
| 기존 FAQ 없음 | "전량 A 지식 부재" 모드로 진행 + 온보딩 성격임을 리포트에 명시 |
| 갭 판단 불확실 | confidence=낮음 + "사람 검토 필요" 큐로 분리, FAQ 초안 자동 생성 안 함 |
| 답변 근거 없음(agent_answer 공란) | 초안에 **[정책 확인 필요]** 마커 — 플러그인이 정책을 지어내지 않음 |
| 재시뮬레이션 개선 없음 | 그대로 보고 (개선 0도 유효한 결과) + 원인 가설(갭 유형 분포) 제시 |
| 로그에 개인정보(전화/이메일) | 적재 단계에서 정규식 마스킹 후 처리 |
| 로그 과다(수천 건+) | 배치 분할 + 샘플링 사용 시 그 사실과 표본 크기를 리포트에 명시 |

## 확정 피치 — S2 "resolution-lift" (하이브리드)

**클러치 피처 한 문장**: *미해결 상담 로그를 넣으면, 지식 갭 진단 → FAQ·규칙 초안 → 재시뮬레이션 검증 리포트까지 — CX 담당자가 ~2주 걸리는 'AI 상담 해결률 개선 컨설팅 1사이클'을 한 번의 실행으로 완주하는 Codex 플러그인.* (문항 1 초안)

**왜 이 문제를 이 방식이 가장 잘 푸는가**:
- 출제자가 명시한 유일한 현재 문제 "이커머스 고객사의 상담 문제"[**01:28**, E02]에서, 공개된 최고 사례(해결률 80%[E12])조차 **최소 20%가 미해결**로 남고, ALF는 등록된 지식 기반으로만 답하므로[E17] 해결률의 레버는 지식 보강뿐이다 — 그런데 그 사이클(로그 읽기→갭 찾기→FAQ 쓰기→검증)이 수작업이다[E16, E36].
- S2는 이 노동을 힌트 영상의 분업론 그대로 나눈다: "AI가 작업은 다 해주고 사람이 하는 건 생각"[**02:16**, E03] — 반복 작업(집계·조립)은 스크립트, 판단(갭 유형·초안·채점)은 LLM, 그리고 **사람에게는 C 정책 결정만** 남긴다. 판단 기준과 프롬프트가 SKILL.md에 그대로 드러나 "프롬프팅·접근을 본다"[**02:12**]는 평가 관점과 정합.
- 파일 완결 파이프라인이라 3,000여 고객사[E14] 어디든 로그만 있으면 반복 적용 — "수천 개, 수만 개 기업의 문제를 푸는 AX 컨설턴트"[**03:47**, E06]의 도구다.

**데모 계획** (문항 5·검증 로그와 정합):
1. **정상 실행**: 동봉 합성 로그 60건 → 4단계 완주 → 리포트 3종 + `report.html`(before/after 해결률 막대 — 비주얼 요소). 55%→78% 유형의 개선 수치가 화면에 보임(합성 데이터 기준임을 리포트에 명시).
2. **예외 실행 A**: 필수 컬럼 빠진 CSV → 템플릿 안내 후 중단하는 모습.
3. **예외 실행 B**: `agent_answer` 없는 갭 → [정책 확인 필요] 마커로 초안 자동 확정을 거부하는 모습.
- 세 실행 모두 이 폴더 세션에서 수행해 `logs/`에 자동 기록 (플러그인↔로그↔question.md 정합).

> ⛔ **사용자 승인 대기** — 승인 전 후속 작업(구현) 시작 금지.
