===== 문항 1 (672자) =====
resolution-lift는 미해결 상담 로그를 넣으면 지식 갭 진단→FAQ·규칙 초안→재시뮬레이션 검증까지, 'AI 상담 해결률 개선 컨설팅' 1사이클을 한 번의 실행으로 완주하는 Codex 플러그인입니다.

채널톡의 상담 AI 에이전트 ALF는 3,000여 기업에서 누적 650만 건의 상담을 처리했지만, 공식 발표된 최고치인 베타 해결률 80%에서도 최소 20%는 AI가 못 풀고 상담사에게 넘어갑니다. ALF는 고객 정보를 학습하지 않으며 등록된 FAQ 기반으로 답하기 때문에, 해결률을 올리는 가장 큰 레버는 지식 보강입니다. 그런데 그 사이클—미해결 로그 수천 건을 읽고, 패턴을 분류하고, 빠진 지식을 찾아 FAQ를 쓰고, 효과를 검증하는 일—은 전부 사람 몫입니다.

사용자는 두 부류입니다. ①ALF를 운영하는 이커머스 고객사의 CX 담당자—소수 인원이 대량 문의를 감당하는 팀이라, 행사 직후 문의가 폭증했는데 무엇부터 FAQ로 만들어야 할지 로그를 열어볼 엄두가 안 나는 상황을 겪습니다. ②그 개선 사이클을 수천 고객사에 반복 지원해야 하는 채널톡 쪽 AX 컨설턴트. 이들이 해결률 정체를 느낀 시점에 상담 로그 CSV와 기존 FAQ 파일 두 개만 들고 이 플러그인을 실행하면, 등록 가능한 FAQ 초안과 '사람이 정해야 할 정책 목록', 그리고 개선 기대치 리포트를 받습니다.

===== 문항 2 (677자) =====
출제 영상에서 채널톡이 "지금 풀고 있는 문제"로 유일하게 명시한 것이 이커머스 고객사의 상담 문제였습니다(1:28). 저는 그 문제의 성과 지표인 'AI 상담 해결률'이 정확히 어디서 막히는지를 공개 자료로 팠습니다.

공개 사례의 해결률은 안다르 61%, 라룸 배송 문의 55%, 비공개 베타 최고 80%—최고 사례조차 최소 20%는 미해결로 남습니다. 원인은 구조적입니다. ALF는 등록된 지식으로만 답하는데(공식 FAQ 설정 가이드), 어떤 지식이 빠졌는지는 미해결 로그를 사람이 읽어야만 보입니다. 자료가 사전 정리된 최상 조건에서도 FAQ 65개 등록에 2.5시간이 걸렸다는 공식 사례가 있고, 그 앞단인 '수천 건 로그에서 갭 찾기'는 훨씬 큽니다. 신규 도입 데모는 'AI 상담 시뮬레이션'(2026-05 출시)으로 회사가 이미 자동화했지만, 도입 후 운영 단계의 이 개선 사이클을 자동화했다는 공개 발표는 찾지 못했습니다(2026-07-09 검색 기준). 여기가 CX 담당자가 실제로 막히는 지점이자, 3,000여 고객사마다 반복되는 병목입니다.

결정적으로, 영상의 "AI가 작업은 다 해주고 사람이 하는 건 생각"(2:16)이라는 관점에서—그 '생각'(갭 판단·초안·채점)마저 구조화해서 사람에게는 정책 결정만 남기는 것. 이것이 채널톡이 말하는 AX와 가장 정합하는 문제라고 판단했습니다.

===== 문항 3 (996자) =====
분업 원칙이 설계의 핵심입니다: 세는 일(집계·검증·리포트)은 결정적 스크립트가, 생각하는 일(분류·초안·채점)은 LLM이, 정책 결정은 사람이 합니다. LLM의 판단 기준은 전부 SKILL.md에 명문화되어 있습니다.

절차는 4단계입니다. ①적재·검증(scripts/load.py): 상담 로그·FAQ CSV의 스키마를 검사하고, 개인정보를 자동 마스킹하고, 해결/미해결을 분리해 기준 해결률을 집계합니다. ②갭 분류(LLM): 미해결 티켓을 클러스터로 묶고 유형을 판정합니다—A 지식 부재/B 지식 불명확(FAQ는 있으나 예외 조건 누락)/C 정책 결정 필요/D 태스크 필요(지식으로 해결 불가)/REVIEW 사람 검토. 판정마다 confidence와 근거 티켓을 남기고, 확신 없으면 짐작하지 않고 REVIEW로 보냅니다. ③초안 생성(LLM): A·B 갭에만 FAQ 초안을 씁니다. 1차 근거는 반드시 상담사 실답변이고, 근거가 없으면 [정책 확인 필요] 마커를 달아 등록을 보류합니다—회사 정책을 지어내지 않습니다. ④재시뮬레이션(LLM 채점→scripts/simulate.py): "초안이 등록됐다면 해결됐을까"를 전건 채점한 뒤, 스크립트가 스키마·환각 티켓 ID·전건 커버를 검증하고 갭 리포트, FAQ 초안, before/after 검증 리포트, HTML 대시보드를 생성합니다.

정보가 부족하거나 안 풀릴 때: 파일이 없거나 필수 컬럼이 빠지면 템플릿을 안내하고 중단합니다(짐작 진행 금지). resolved 컬럼이 없으면 상담사 답변 유무로 추정하되 모든 리포트에 (추정)을 표기합니다. FAQ가 비어 있으면(헤더는 필수) 온보딩 성격임을 명시하고 진행합니다. 상담사 답변이 서로 모순인 클러스터는 REVIEW로 재분류합니다. 시뮬레이션 개선이 0이어도 그대로 보고하고 원인 가설을 제시합니다. 200건 초과 시 50건씩 배치 처리합니다. '부분' 판정은 보수적으로 미해결로 집계해 개선치를 부풀리지 않습니다. 세부는 zip의 README.md와 동일합니다.

===== 문항 4 (731자) =====
이 제출물은 리서치부터 코드·검증까지 AI(Claude Code·Codex, 로그 동봉)가 실행하고, 사람은 방향 결정과 승인 게이트를 맡는 구조로 만들었습니다.

AI에게 맡긴 것: 출제 영상 전사와 의도 구조화(사실 F1~F9/추정 I1~I7 분리), 공개 근거 40건의 대장화와 URL 전수 접근성 검증, 문제·솔루션 후보 비교, 플러그인 코드(스크립트 2개·SKILL.md)와 합성 데이터 60건 제작, 구동 검증.

직접 판단한 것: ①문제 선정 — AI가 비교한 후보 3개 중 '미해결 상담 지식 보강 사이클'을 승인 ②수치 프레이밍 — AI 초안의 '미해결 20~45%'는 해결률의 여집합에서 파생된 추정이라 대표 수치로 쓰지 못하게 하고 '최고 사례 80%에서도 최소 20%'라는 하한 서술로 강등 지시 ③솔루션 형태 — 순수 프롬프트형과 API 연동형 대신 하이브리드 승인.

막힌 지점과 해결: 리서치 중 API 오류로 턴 중단 → 완료 단계를 점검해 step-03부터 재개(세션 로그에 기록). 검색 요약에 있던 '일본 1.5만 기업' 수치가 기사 원문에 없음을 URL 검증에서 발견 → 삭제. load.py의 --out 인자 파싱 버그 → argparse로 교체.

받아들이지 않은 AI 제안: 신규 고객사 온보딩 자동화 후보는 유망해 보였지만, 채널톡이 'AI 상담 시뮬레이션'(2026-05)으로 이미 자동화에 착수한 정면 영역이라 기각하고 도입 후 운영 단계의 공백을 택했습니다.

===== 문항 5 (796자) =====
검증은 심사자 절차 그대로 수행하고 전 과정을 세션 로그에 남겼습니다(logs/claude-code, logs/codex).

정상: 로컬 marketplace에 등록(codex plugin marketplace add → plugin add)한 뒤 Codex CLI에서 스킬 실행을 지시했습니다. 입력 tickets.csv 60건·faq.csv 12건 → 미해결 27건이 7개 클러스터로 분류되고(상품케어A9·쿠폰조건B7·해외배송B4·환불안내A2·환불처리D2·제휴할인C2·감정REVIEW1), A·B에만 초안 4건 생성, 재시뮬레이션 해결률 55.0%→85.0%(+18건). C·D·REVIEW는 '사람 몫' 목록으로 분리되고 '부분' 5건은 미해결로 보수 집계됨을 확인했습니다.

예외: ①question 컬럼 없는 CSV → 필수 컬럼 안내·템플릿 출력 후 중단(출력 미생성, 짐작 진행 없음) ②시뮬레이션 JSON의 티켓 ID 변조 → 환각 ID로 검출해 리포트 생성 거부.

의심한 것: LLM이 관대하게 채점해 개선치가 부풀 위험 → '부분'을 미해결로 집계하고 리포트에 '시뮬레이션 판정값, 실측 아님' 고지를 자동 포함.

고친 것: load.py 인자 파싱 버그(argparse 교체), 공개 문서와 다른 marketplace 매니페스트 위치(.agents/plugins/marketplace.json)를 실행으로 확인해 README에 반영.

부족한 것: 실제 채널톡 수출 로그로는 미검증(합성 데이터). LLM 단계는 Codex가, 마지막 스크립트 단계는 Claude Code 세션이 완결(로그 동봉).
